# SS2 Project

Habit-tracking monorepo with:

- `Frontend/`: Expo Router + React Native mobile app
- `Backend/`: Express API server
- Supabase for auth and persistence

## Product Snapshot

Current app scope includes:

- onboarding flow
- email + OAuth authentication
- home dashboard
- habit creation and management
- time-based focus mode for duration habits
- streak / EXP / HP / level progression
- analytics dashboard
- floating AI coach bubble

## Quick Start

1. Install dependencies:

```bash
cd Backend
npm install

cd ../Frontend
npm install
```

2. Create environment files.

Backend: `Backend/.env`

```env
PORT=4000
SUPABASE_URL=your_supabase_project_url
SUPABASE_SERVICE_ROLE_KEY=your_supabase_service_role_key
```

Frontend: `Frontend/.env`

```env
EXPO_PUBLIC_SUPABASE_URL=your_supabase_project_url
EXPO_PUBLIC_SUPABASE_ANON_KEY=your_supabase_anon_key
```

3. Start the backend:

```bash
cd Backend
npm run dev
```

Health check:

```txt
http://localhost:4000/api/health
```

4. Start the frontend:

```bash
cd Frontend
npm start
```

`npm start` is recommended here because it runs `configure:lan` first and refreshes `Frontend/.env.local` with your current LAN IP before Expo starts.

## Frontend Start Modes

Recommended commands:

```bash
npm start
npm run android:lan
npm run android:emulator
npm run start:emulator
npm run lint
```

Important behavior:

- `npm start` runs `configure:lan && expo start`
- `npm run android:lan` runs `configure:lan` before launching Android
- `npm run android:emulator` runs `configure:emulator` before launching Android
- direct commands such as `npx expo start -c`, `npm run android`, `npm run ios`, and `npm run web` do not refresh `Frontend/.env.local` automatically

If your Wi-Fi IP changes and the frontend suddenly cannot reach the backend, rerun:

```bash
cd Frontend
npm run configure:lan
```

## Backend URL Resolution

Frontend backend resolution lives in `Frontend/src/services/api.js`.

It supports:

- `EXPO_PUBLIC_API_URL`
- `EXPO_PUBLIC_API_HOST` + `EXPO_PUBLIC_API_PORT` + `EXPO_PUBLIC_API_SCHEME`
- Expo/LAN host inference
- Android emulator aliases `10.0.2.2` and `10.0.3.2`

The frontend probes `/api/health` and uses the first healthy backend candidate.

`Frontend/.env.local` is machine-specific, auto-generated by the frontend scripts, and should stay out of git.

## Supabase OAuth Notes

The app declares Expo scheme `project` in `Frontend/app.json`, but the actual redirect URL depends on runtime:

- development build / standalone app: typically `project://auth-callback`
- Expo Go: typically an `exp://.../--/auth-callback` URL

Current frontend OAuth logic lives in `Frontend/src/services/supabaseAuth.js` and now uses `expo-auth-session` to generate the redirect URL more reliably for Expo Go.

Supabase configuration guidance:

- `Site URL`: `project://auth-callback`
- `Redirect URLs`: add the exact runtime URL printed by the app during OAuth in Expo Go, or a matching wildcard pattern if your Supabase setup supports it safely

If OAuth works in a build but not in Expo Go, the most common cause is that Supabase `Redirect URLs` do not match the real `exp://...` redirect URL.

## Recent Performance Work

The analytics screen was recently optimized:

- backend now supports bundled analytics periods in one request
- frontend caches analytics responses by period
- analytics screen loads only the periods it needs first, then lazy-loads additional ranges

This reduced repeated backend work and improved first-load behavior on the analytics tab.

## Regression Checklist

After significant changes, verify:

- backend starts and `/api/health` responds
- frontend can connect to backend on LAN or emulator
- email auth still works
- OAuth returns to `auth-callback`
- creating a normal habit still works
- creating a time-based habit still opens focus mode
- normal habit can complete and undo
- analytics tab still loads and range switching still works
- home still shows all habits
- coach bubble still opens and can be dragged
