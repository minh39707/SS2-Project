# HabitForge Frontend

Frontend mobile app for HabitForge, built with Expo Router and React Native.

This app covers:
- onboarding flow for choosing a habit and schedule
- authentication with email and OAuth
- Supabase session handling on mobile
- dashboard experience for the signed-in user

## Tech Stack

- Expo SDK 54
- React Native 0.81
- Expo Router
- Supabase Auth
- AsyncStorage

## Supported Auth Methods

- Email / password
- Google
- Facebook
- GitHub

OAuth is handled with Supabase plus `expo-web-browser` and a mobile callback route at `auth-callback`.

## What Is `.env.local`?

`Frontend/.env.local` is a local, machine-specific override file for frontend runtime settings.

In this project it is mainly used for backend connection settings such as:

```env
EXPO_PUBLIC_API_URL=http://192.168.1.10:4000/api
EXPO_PUBLIC_API_HOST=192.168.1.10
EXPO_PUBLIC_API_PORT=4000
EXPO_PUBLIC_API_SCHEME=http
```

Do you need it?

- `Yes`, in normal local development for this repo it is usually useful and often auto-generated by the provided scripts.
- `Yes`, if your phone or emulator needs a machine-specific backend host.
- `No`, if your current setup already works through other env values and you are not overriding the backend address.

How this repo uses it:

- `npm start` runs `configure:lan` first and refreshes `Frontend/.env.local`
- `npm run android:lan` also refreshes `Frontend/.env.local`
- `npm run android:emulator` writes emulator-friendly values into `Frontend/.env.local`

Important notes:

- `.env.local` is ignored by git in this repo and should stay local to your machine
- use `Frontend/.env` for shared frontend config
- use `Frontend/.env.local` for machine-specific overrides only
- if your Wi-Fi or LAN IP changes, rerun `npm run configure:lan`

## Backend Requirement

This frontend expects the local backend to be running on port `4000`.

From the `Backend` folder:

```bash
npm install
npm run dev
```

Default backend URL assumptions in the app:
- Expo device on LAN: `http://<your-dev-machine-ip>:4000/api`
- Android emulator: `http://10.0.2.2:4000/api`
- iOS simulator / local web: `http://localhost:4000/api`

Recommended startup order:

1. Start the backend
2. Start the frontend
3. If you change `.env` or `.env.local`, restart Expo with `npx expo start -c`

Recommended setup for multiple devices on the same Wi-Fi:
- Run `npm run start:lan` to auto-detect your current LAN IP and write it to `.env.local`.
- This generated `.env.local` is ignored by git, so each developer or machine can keep its own network config.
- Keep the backend running on your dev machine at port `4000`.
- Only set `EXPO_PUBLIC_API_URL` or `EXPO_PUBLIC_API_HOST` when you need to point every device to a fixed backend host manually.

Important script behavior:

- `npm start` and `npm run start:lan` refresh `.env.local` automatically.
- `npm run android:lan` refreshes `.env.local` automatically.
- `npm run android:emulator` writes the Android emulator alias `10.0.2.2`.
- `npm run android`, `npm run ios`, `npm run web`, and direct `npx expo start -c` do not refresh `.env.local` first.

If backend connectivity suddenly fails after switching Wi-Fi or IP address:

```bash
cd Frontend
npm run configure:lan
```

Example:

```bash
cd Frontend
npm run start:lan
```

Or, if Expo is already running:

```bash
npm run configure:lan
```

## Supabase Auth URL Configuration

This app declares Expo scheme `project`, but the final redirect URL depends on runtime:

```txt
Development build / standalone app: project://auth-callback
Expo Go: exp://<lan-ip>:<expo-port>/--/auth-callback
```

Current OAuth redirect creation lives in `src/services/supabaseAuth.js` and uses `expo-auth-session`.

Recommended Supabase setup:

```txt
Site URL: project://auth-callback
```

Also add the exact runtime redirect URL shown in the frontend OAuth debug log to `Redirect URLs` when testing with Expo Go. If you rely on wildcard redirect patterns, verify that they really match the generated `exp://.../--/auth-callback` URL.

## Current App Behavior

- onboarding state is stored locally per account scope
- signed-in sessions are persisted with Supabase
- onboarding habit sync is sent to the backend after authentication when needed
- dashboard requests use the authenticated access token
- analytics responses are cached by period and the analytics screen lazy-loads unused ranges

## Troubleshooting

If OAuth seems stuck or returns to the wrong place:

- make sure the backend is running
- confirm `.env` has valid Supabase values
- confirm Supabase `Site URL` is `project://auth-callback`
- if using Expo Go, confirm Supabase `Redirect URLs` includes the real `exp://.../--/auth-callback` URL printed by the app
- restart Metro with cache clear:

```bash
npx expo start -c
```

If the app cannot reach the API, verify that your phone and dev machine are on the same network and that port `4000` is reachable.

If you switch between emulator and real devices often:
- Prefer `npm run start:lan` for Expo Go on LAN.
- Prefer `npm run android:emulator` or `npm run start:emulator` for Android emulator sessions.
- Use `EXPO_PUBLIC_API_URL=http://10.0.2.2:4000/api` only when you intentionally want to pin the emulator to a fixed backend URL.

If you see the Expo Go Android warning about remote push notifications from `expo-notifications`, that is an Expo Go limitation rather than a local app bug. Local habit reminders may still work, but full remote push support requires a development build.
